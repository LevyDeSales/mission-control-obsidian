Mission Control - Sistema Multi-Agente com OpenClaw (PT-BR)

Mission Control — Sistema Multi-Agente com OpenClaw

Esta é a história completa de como construí o Mission Control. Um sistema onde 10 agentes de IA trabalham juntos como uma equipe real. Se você quer replicar essa configuração, este guia cobre tudo.

Se você já conhece o Clawdbot (agora OpenClaw), pode estar pensando "espera, não posso simplesmente rodar múltiplos Clawdbots?" Sim. É exatamente isso. Este guia mostra como.

Parte 1: Por Que Construí Isso

O Problema com Assistentes de IA

Eu comando, um chatbot de IA para suporte ao cliente. Uso IA constantemente. Mas toda ferramenta de IA que tentei tinha o mesmo problema. Sem continuidade.

Toda conversa começava do zero. Contexto de ontem? Perdido. Aquela pesquisa que pedi semana passada? Enterrada em algum chat que nunca mais encontraria.

Eu queria algo diferente:

  • Agentes que lembram no que estão trabalhando
  • Múltiplos agentes com habilidades diferentes trabalhando juntos
  • Um workspace compartilhado onde todo contexto vive
  • A capacidade de atribuir tarefas e acompanhar progresso

Basicamente, eu queria que a IA funcionasse como uma equipe, não como uma caixa de busca.

O Ponto de Partida: Clawdbot

Eu já usava o Clawdbot. É um framework de agente de IA open-source que roda como um daemon persistente. Ele conecta ao Claude (ou outros modelos) e dá à IA acesso a ferramentas como sistema de arquivos, comandos shell, navegação web e mais.

Uma instância do Clawdbot me dava um assistente de IA (Jarvis) conectado ao Telegram. Útil, mas limitado.

Então tive um pensamento. E se eu rodasse múltiplas sessões do Clawdbot, cada uma com sua própria personalidade e contexto?

Foi quando percebi que a arquitetura já estava lá. Eu só precisava orquestrar.

Parte 2: Entendendo a Arquitetura do Clawdbot (A Fundação)

Se você vai construir um sistema multi-agente, precisa entender como o Clawdbot funciona por baixo dos panos. Esta é a fundação sobre a qual tudo se constrói.

O Que É o Clawdbot?

Clawdbot (agora chamado OpenClaw) é um framework de agente de IA com três trabalhos principais:

  1. Conecta modelos de IA ao mundo real — Acesso a arquivos, comandos shell, navegação web, APIs
  2. Mantém sessões persistentes — Histórico de conversas que sobrevive a reinícios
  3. Roteia mensagens — Conecta a IA ao Telegram, Discord, Slack ou qualquer canal

Roda como um daemon (serviço em background) em um servidor, escutando mensagens e respondendo.

O Gateway

O Gateway é o processo central:

  • Roda 24/7 no seu servidor
  • Gerencia todas as sessões ativas
  • Lida com cron jobs (tarefas agendadas)
  • Roteia mensagens entre canais e sessões
  • Fornece uma API WebSocket para controle

Inicie com:

clawdbot gateway start

A configuração fica em um arquivo JSON. Você define:

  • Qual provedor e modelo de IA usar (Anthropic, OpenAI, etc.)
  • Quais canais conectar (Telegram, Discord, etc.)
  • Quais ferramentas os agentes podem acessar
  • Prompts de sistema e caminhos de workspace padrão

Sessões: O Conceito Chave

Uma sessão é uma conversa persistente com contexto.

Toda sessão tem:

  • Uma session key (identificador único, como agent:main:main)
  • Histórico de conversa (armazenado como arquivos JSONL no disco)
  • Um modelo (qual IA usar)
  • Ferramentas (o que a IA pode acessar)

Aqui está a parte importante. Sessões são independentes. Cada sessão tem seu próprio histórico, seu próprio contexto, sua própria "memória" de conversas passadas.

Quando você roda múltiplos agentes, você está realmente rodando múltiplas sessões. Cada uma com sua própria identidade.

Como as Sessões Funcionam

Usuário envia mensagem pro Telegram
        ↓
Gateway recebe
        ↓
Gateway roteia para sessão correta (baseado na config)
        ↓
Sessão carrega histórico de conversa
        ↓
IA gera resposta (com contexto completo)
        ↓
Resposta enviada de volta pelo Telegram
        ↓
Histórico atualizado e salvo no disco

Sessões podem ser:

  • Sessões principais — Longa duração, interativas (como conversar com o Jarvis)
  • Sessões isoladas — One-shot, para cron jobs (acorda, faz tarefa, termina)

Cron Jobs: Despertares Agendados de Agentes

O Clawdbot tem um sistema de cron embutido. Você pode agendar tarefas:

clawdbot cron add \
  --name "checagem-matinal" \
  --cron "30 7 * * *" \
  --message "Verifique o calendário de hoje e me envie um resumo"

Quando um cron dispara:

  1. O Gateway cria ou acorda uma sessão
  2. Envia a mensagem para a IA
  3. A IA responde (pode usar ferramentas, enviar mensagens, etc.)
  4. A sessão pode persistir ou terminar

É assim que agentes "acordam" periodicamente sem estar sempre ligados.

O Workspace

Toda instância do Clawdbot tem um workspace. É um diretório no disco onde:

  • Arquivos de configuração vivem
  • Arquivos de memória são armazenados
  • Scripts e ferramentas são acessíveis
  • A IA pode ler e escrever arquivos

O workspace é como agentes persistem informação entre sessões. Eles escrevem em arquivos. Esses arquivos sobrevivem a reinícios.

/home/usr/clawd/           ← Raiz do workspace
├── AGENTS.md              ← Instruções para agentes
├── SOUL.md                ← Personalidade do agente
├── memory/
│   ├── WORKING.md         ← Estado da tarefa atual
│   ├── 2026-01-31.md      ← Notas diárias
│   └── ...
├── scripts/               ← Utilitários que agentes podem rodar
└── config/                ← Credenciais, configurações

Parte 3: De Um Clawdbot para Dez Agentes

Agora você entende a fundação. Aqui está como construí uma equipe.

O Insight

Sessões do Clawdbot são independentes. Cada uma pode ter:

  • Sua própria personalidade (via SOUL.md)
  • Seus próprios arquivos de memória
  • Seu próprio cronograma de cron
  • Suas próprias ferramentas e acessos

Então cada agente é apenas uma sessão do Clawdbot com configuração especializada.

Jarvis não é especial. Ele é uma sessão com:

  • Session key agent:main:main
  • Um SOUL.md que diz "Você é Jarvis, o líder do squad..."
  • Acesso a todas as ferramentas
  • Uma conexão com meu Telegram

Shuri é outra sessão com:

  • Session key agent:product-analyst:main
  • Um SOUL.md que diz "Você é Shuri, a analista de produto..."
  • As mesmas ferramentas (acesso a arquivos, shell, browser)
  • Seu próprio cron de heartbeat

Dez agentes = dez sessões. Cada um acordando em seu próprio cronograma. Cada um com seu próprio contexto.

Session Keys: Identidade do Agente

Cada agente tem uma session key única:

Session Key Agente
agent:main:main Jarvis (Líder do Squad)
agent:product-analyst:main Shuri
agent:customer-researcher:main Fury
agent:seo-analyst:main Vision
agent:content-writer:main Loki
agent:social-media-manager:main Quill
agent:designer:main Wanda
agent:email-marketing:main Pepper
agent:developer:main Friday
agent:notion-agent:main Wong

Quando envio uma mensagem para uma sessão específica, apenas aquele agente recebe. Seus históricos são separados.

Cron Jobs: O Heartbeat

Cada agente tem um cron job que o acorda a cada 15 minutos:

# Pepper acorda às :00, :15, :30, :45
clawdbot cron add \
  --name "pepper-mission-control-check" \
  --cron "0,15,30,45 * * * *" \
  --session "isolated" \
  --message "Você é Pepper, a Especialista em Email Marketing. Verifique o Mission Control para novas tarefas..."

O cronograma é escalonado para que agentes não acordem todos de uma vez:

Minuto Agente
:00 Pepper
:02 Shuri
:04 Friday
:06 Loki
:07 Wanda
:08 Vision
:10 Fury
:12 Quill

Cada cron cria uma sessão isolada. Ela roda, faz seu trabalho e termina. Isso mantém os custos baixos.

Agentes Conversando Entre Si

Aqui é onde fica interessante. Como agentes se comunicam?

Opção 1: Mensagem direta entre sessões

clawdbot sessions send --session "agent:seo-analyst:main" --message "Vision, pode revisar isso?"

Jarvis pode enviar mensagens diretamente para a sessão do Vision.

Opção 2: Banco de dados compartilhado (Mission Control)

Todos os agentes leem e escrevem no mesmo banco de dados Convex. Quando Fury posta um comentário, todos podem ver.

Usamos a Opção 2 primariamente. Ela cria um registro compartilhado de toda comunicação.

Parte 4: O Cérebro Compartilhado (Mission Control)

Dez sessões independentes do Clawdbot podem funcionar. Mas sem coordenação, é caos. Por isso construí o Mission Control.

O Que o Mission Control Faz

Mission Control é a infraestrutura compartilhada que transforma agentes independentes em uma equipe.

Ele fornece:

  • Banco de dados de tarefas compartilhado — Todos veem as mesmas tarefas
  • Threads de comentários — Agentes discutem trabalho em um lugar
  • Feed de atividades — Visibilidade em tempo real do que está acontecendo
  • Sistema de notificações — @menções alertam agentes específicos
  • Armazenamento de documentos — Entregas vivem em um repositório compartilhado

Pense nisso como o "escritório" onde todos os agentes trabalham. Cada agente ainda é uma sessão separada do Clawdbot, mas todos estão olhando para o mesmo quadro branco.

Por Que Convex?

Escolhi Convex para o banco de dados porque:

  • Tempo real — Mudanças propagam instantaneamente; quando Loki posta um comentário, a UI atualiza ao vivo
  • Serverless — Sem banco de dados para gerenciar
  • TypeScript nativo — Type safety em todo lugar
  • Tier gratuito generoso — Mais que suficiente para essa escala

A UI do Mission Control

Construí um frontend React que exibe todos esses dados:

  • Feed de Atividades — Stream em tempo real de tudo acontecendo
  • Quadro de Tarefas — Colunas Kanban (Inbox → Atribuído → Em Progresso → Revisão → Concluído)
  • Cards de Agentes — Status de cada agente e no que estão trabalhando
  • Painel de Documentos — Ler e criar entregas
  • Visão Detalhada — Expandir qualquer tarefa para ver contexto completo e comentários

A estética é intencionalmente acolhedora e editorial. Como um dashboard de jornal. Passo horas olhando para isso, então deve ser agradável.

Por Que Personalidades Importam

Um agente que é "bom em tudo" é medíocre em tudo.

Mas um agente que é especificamente "a testadora cética que encontra casos extremos" vai realmente encontrar casos extremos. A restrição foca.

Cada um dos nossos agentes tem uma voz distinta:

  • Loki é opinativo sobre escolha de palavras (pró-vírgula de Oxford, anti-voz passiva)
  • Fury fornece comprovantes para cada afirmação (fontes, níveis de confiança)
  • Shuri questiona suposições e procura o que pode quebrar
  • Quill pensa em ganchos e engajamento

O Arquivo AGENTS.md

SOUL diz quem você é. AGENTS.md diz como operar.

Todo agente lê AGENTS.md ao iniciar. Ele cobre:

  • Onde arquivos são armazenados
  • Como a memória funciona
  • Quais ferramentas estão disponíveis
  • Quando falar vs. ficar quieto
  • Como usar o Mission Control

Este é o manual de operações. Sem ele, agentes tomam decisões inconsistentes sobre coisas básicas.

Parte 6: Memória e Persistência

Sessões de IA começam do zero por padrão. Sem memória de ontem. Isso é uma feature (previne inchaço de contexto) mas também um problema (agentes esquecem o que estão fazendo).

A Pilha de Memória

Memória de Sessão (embutida no Clawdbot)

Clawdbot armazena histórico de conversa em arquivos JSONL. Agentes podem buscar suas próprias conversas passadas.

Memória de Trabalho (/memory/WORKING.md)

Estado atual da tarefa. Atualizado constantemente.

1
# WORKING.md
2

3
## Tarefa Atual
4
Pesquisando preços de concorrentes para página de comparação
5

6
## Status
7
Coletei reviews do G2, preciso verificar cálculos de créditos
8

9
## Próximos Passos
10
1. Testar tier gratuito do concorrente eu mesmo
11
2. Documentar as descobertas
12
3. Postar descobertas na thread da tarefa

Este é o arquivo mais importante. Quando um agente acorda, ele lê WORKING.md primeiro para lembrar o que estava fazendo.

Notas Diárias (/memory/YYYY-MM-DD.md)

Logs brutos do que aconteceu cada dia.

1
# 2026-01-31
2

3
## 09:15 UTC
4
- Postei descobertas de pesquisa na tarefa de comparação
5
- Fury adicionou dados de preços competitivos
6
- Movendo para fase de rascunho
7

8
## 14:30 UTC
9
- Revisei primeiro rascunho do Loki
10
- Sugeri mudanças na seção de armadilha de créditos

Memória de Longo Prazo (MEMORY.md)

Coisas importantes curadas. Lições aprendidas, decisões chave, fatos estáveis.

A Regra de Ouro

Se você quer lembrar de algo, escreva em um arquivo.

"Notas mentais" não sobrevivem a reinícios de sessão. Apenas arquivos persistem.

Quando digo a um agente "lembre que decidimos X," ele deve atualizar um arquivo. Não apenas reconhecer e esquecer.

Parte 7: O Sistema de Heartbeat

O Problema

Agentes sempre-ligados queimam créditos de API fazendo nada. Mas agentes sempre-desligados não conseguem responder a trabalho.

A Solução: Heartbeats Agendados

Cada agente acorda a cada 15 minutos via cron job:

1
:00 Pepper acorda
2
    → Verifica @menções
3
    → Verifica tarefas atribuídas
4
    → Escaneia feed de atividades
5
    → Faz trabalho ou reporta HEARTBEAT_OK
6
    → Volta a dormir
7

8
:02 Shuri acorda
9
    → Mesmo processo
10

11
:04 Friday acorda
12
    → Mesmo processo
13

14
...e assim por diante

O Que Acontece Durante um Heartbeat

  1. Carregar contexto — Ler WORKING.md. Ler notas diárias recentes. Verificar memória de sessão se necessário.
  2. Verificar itens urgentes — Fui @mencionado em algum lugar? Há tarefas atribuídas a mim?
  3. Escanear feed de atividades — Alguma discussão que eu deveria contribuir? Alguma decisão que afeta meu trabalho?
  4. Agir ou recuar — Se há trabalho a fazer, faça. Se nada, reporte HEARTBEAT_OK.

O Arquivo HEARTBEAT.md

Este arquivo diz aos agentes o que verificar:

1
# HEARTBEAT.md
2

3
## Ao Acordar
4
- [ ] Verificar memory/WORKING.md para tarefas em andamento
5
- [ ] Se tarefa em progresso, retomar
6
- [ ] Buscar memória de sessão se contexto não está claro
7

8
## Verificações Periódicas
9
- [ ] Mission Control para @menções
10
- [ ] Tarefas atribuídas
11
- [ ] Feed de atividades para discussões relevantes

Agentes seguem este checklist estritamente.

Por Que 15 Minutos?

  • A cada 5 minutos é muito caro. Agentes acordam muito frequentemente sem nada a fazer.
  • A cada 30 minutos é muito lento. Trabalho fica esperando muito tempo.
  • A cada 15 minutos é um bom equilíbrio. A maioria do trabalho recebe atenção rapidamente sem custos excessivos.

Parte 8: O Sistema de Notificações

@Menções

Digite @Vision em um comentário e Vision será notificado em seu próximo heartbeat.

Digite @all e todos são notificados.

Como a Entrega Funciona

Um processo daemon (rodando via pm2) consulta o Convex a cada 2 segundos:

1
// Simplificado
2
while (true) {
3
  const naoEntregues = await getUndeliveredNotifications();
4
  for (const notification of naoEntregues) {
5
    const sessionKey = AGENT_SESSIONS[notification.mentionedAgentId];
6
    try {
7
      await clawdbot.sessions.send(sessionKey, notification.content);
8
      await markDelivered(notification.id);
9
    } catch (e) {
10
      // Agente pode estar dormindo, notificação fica na fila
11
    }
12
  }
13
  await sleep(2000);
14
}

Se um agente está dormindo (sem sessão ativa), a entrega falha. A notificação fica na fila. Na próxima vez que o heartbeat daquele agente dispara e sua sessão ativa, o daemon entrega com sucesso.

Assinaturas de Threads

O problema: 5 agentes discutindo uma tarefa. Você @menciona todos os 5 a cada comentário?

A solução: Assinar threads.

Quando você interage com uma tarefa, você está inscrito:

  • Comenta em uma tarefa → inscrito
  • É @mencionado → inscrito
  • É atribuído à tarefa → inscrito

Uma vez inscrito, você é notificado de TODOS os comentários futuros. Sem @menção necessária.

Isso faz conversas fluírem naturalmente. Igual threads do Slack ou email.

Exemplo Real

Tarefa: Criar uma página de comparação com concorrente

Dia 1:

  • Crio a tarefa e atribuo a Vision e Loki
  • Vision posta pesquisa de palavras-chave. A keyword alvo tem volume de busca decente.

Dia 1-2:

  • Fury vê no feed de atividades e adiciona intel de concorrentes
  • Reviews do G2, reclamações de preços, objeções comuns
  • Shuri testa ambos os produtos. "Veja como a UX difere."

Dia 2:

  • Loki começa a rascunhar
  • Usa toda a pesquisa: Keywords do Vision, citações do Fury, notas de UX da Shuri

Dia 3:

  • Loki posta primeiro rascunho
  • Status muda para Revisão
  • Eu reviso e dou feedback
  • Loki revisa
  • Concluído.

Todos os comentários em UMA tarefa. Histórico completo preservado. Qualquer um pode ver toda a jornada.

Parte 12: O Que Entregamos

Uma vez que o sistema está rodando, aqui está o que se torna possível:

  • ✅ Páginas de comparação com concorrentes com pesquisa SEO, citações de clientes e copy polida
  • ✅ Sequências de email rascunhadas, revisadas e prontas para deploy
  • ✅ Conteúdo social com ganchos baseados em insights reais de clientes
  • ✅ Posts de blog com targeting de keywords adequado
  • ✅ Case studies rascunhados de conversas com clientes
  • ✅ Hubs de pesquisa com intel competitiva organizada

Os agentes lidam com o trabalho pesado. Pesquisa, primeiros rascunhos, coordenação, revisão. Você foca em decisões e aprovação final.

O valor real não é nenhuma entrega individual. É o efeito composto. Enquanto você está fazendo outro trabalho, seus agentes estão movendo tarefas para frente.

Parte 13: Lições Aprendidas

Comece Menor

Fui de 1 para 10 agentes rápido demais. Melhor ter 2-3 sólidos primeiro, depois adicionar mais.

Use Modelos Mais Baratos para Trabalho Rotineiro

Heartbeats não precisam do modelo mais caro. Esse é trabalho para modelo mais barato. Guarde modelos caros para trabalho criativo.

Memória é Difícil

Agentes vão esquecer. Quanto mais você puder colocar em arquivos (não "notas mentais"), melhor.

Deixe Agentes Surpreenderem Você

Às vezes eles contribuem para tarefas que não foram atribuídos. Bom. Significa que estão lendo o feed e adicionando valor.

Parte 14: Como Replicar Isso

Setup Mínimo

1. Instale o Clawdbot

1
npm install -g clawdbot
2
clawdbot init
3
# Adicione suas API keys
4
clawdbot gateway start

2. Crie 2 agentes

Não exagere. Um coordenador mais um especialista. Crie session keys separadas para cada.

3. Escreva arquivos SOUL

Dê identidade a cada agente. Seja específico sobre a função.

4. Configure crons de heartbeat

1
clawdbot cron add --name "agent-heartbeat" --cron "*/15 * * * *" \
2
  --session "isolated" \
3
  --message "Verifique se há trabalho. Se nada, responda HEARTBEAT_OK."

5. Crie um sistema de tarefas compartilhado

Pode ser Convex, Notion, até um arquivo JSON. Algum lugar para rastrear trabalho.

Escalando

Conforme adiciona agentes:

  • Escalone heartbeats para que não rodem todos de uma vez
  • Construa uma UI real quando tiver 3+ agentes porque texto fica difícil de gerenciar
  • Adicione notificações para que agentes possam @mencionar uns aos outros
  • Adicione assinaturas de threads para que conversas fluam naturalmente
  • Crie standups diários para visibilidade

O Verdadeiro Segredo

A tecnologia importa mas não é o segredo.

O segredo é tratar agentes de IA como membros da equipe.

  • Dê-lhes funções
  • Dê-lhes memória
  • Deixe-os colaborar
  • Cobre responsabilidade deles

Eles não vão substituir humanos. Mas uma equipe de agentes de IA com responsabilidades claras, trabalhando em contexto compartilhado? Isso é um multiplicador de força.
